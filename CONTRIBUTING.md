## Code of Conduct

This project and everyone participating in it is governed by our Code of Conduct. By participating, you are expected to uphold this code. Please report unacceptable behavior to the project maintainers.

## Getting Started

1. Fork the repository on GitHub
2. Clone your forked repository locally
3. Create a new branch for your contribution
4. Make your changes
5. Test thoroughly
6. Submit a pull request

## How to Contribute

### Areas for Contribution

We welcome contributions in several areas:

- **Game Mechanics**: Improvements to combat, abilities, and tactical systems
- **AI Game Master**: Enhancements to the AI GM engine, narrative generation, and decision-making
- **Documentation**: Improved documentation, guides, and examples
- **Bug Fixes**: Fixes for identified issues
- **Features**: New features that align with SENTINEL's vision
- **Testing**: Unit tests, integration tests, and scenario testing

## Development Setup

1. Ensure you have the required dependencies installed
2. Follow the project's coding standards and conventions
3. Use meaningful commit messages
4. Keep commits focused and atomic
5. Test your changes before submitting

## Submission Guidelines

### Code Style

- Follow the existing code style and conventions
- Write clean, readable code with meaningful variable names
- Add comments for complex logic and algorithms
- Keep functions and methods focused and modular

### Documentation

- Update relevant documentation with your changes
- Add docstrings to new functions and classes
- Include examples where appropriate
- Update the README if your changes affect project usage

### Testing

- Write tests for new features
- Ensure all existing tests pass
- Aim for meaningful test coverage
- Test edge cases and error conditions

## Reporting Bugs

When reporting bugs, please include:

- **Title**: Clear and descriptive bug title
- **Description**: Detailed description of the issue
- **Steps to Reproduce**: Clear steps to reproduce the problem
- **Expected Behavior**: What you expected to happen
- **Actual Behavior**: What actually happened
- **Environment**: OS, version, and relevant system information
- **Screenshots/Logs**: If applicable, include screenshots or error logs

## Suggesting Enhancements

We encourage suggestions for enhancements! When suggesting an enhancement:

- Use a clear and descriptive title
- Provide a step-by-step description of the suggested enhancement
- Provide specific examples to demonstrate the steps
- Describe the current behavior and expected new behavior
- Explain why this enhancement would be useful

## Pull Request Process

1. **Fork and Branch**: Create a feature branch from `master`
2. **Make Changes**: Implement your changes following the guidelines above
3. **Test Thoroughly**: Ensure all tests pass and your changes work as intended
4. **Commit Messages**: Use clear, descriptive commit messages
5. **Push Changes**: Push your branch to your forked repository
6. **Create Pull Request**: Open a pull request against the main repository
7. **PR Description**: Include:
   - Clear description of changes
   - Reference to related issues (if any)
   - Testing performed
   - Any breaking changes
8. **Review Process**: Respond to feedback and review comments
9. **Approval**: Once approved, your PR will be merged

### Pull Request Checklist

- [ ] My code follows the project's style guidelines
- [ ] I have updated relevant documentation
- [ ] I have added tests for my changes
- [ ] All tests pass locally
- [ ] My changes don't break existing functionality
- [ ] I have made corresponding changes to documentation
- [ ] My commit messages are clear and descriptive
